# NexaChat Core Flow and Function Iteration Plan 01

## 计划目标

把当前应用从“页面齐了”推进到“流程清楚、行为明确、结果可验证”。用户需要能顺着固定路径完成：启动应用 -> 确认工作区/模型 -> 发起对话 -> 切换提供商或模型 -> 保留本地历史 -> 查看日志与诊断 -> 完成配置与导入导出。

## 当前问题

- 一些功能区已经出现，但流程边界仍不够清楚，容易把“已实现”和“预留”混在一起理解。
- 页面里存在重复的状态摘要，但缺少对当前任务真正有用的反馈。
- 切换模型、提供商、网关或知识上下文时，用户难以一眼确认哪些信息会保留，哪些会变化。
- 错误、诊断、日志、复制错误、打开日志这类动作没有统一的闭环。
- 预留的 Agent / MCP / 工作流能力需要严格和真实执行能力分开表达。
- 导入导出、备份恢复、快照和冲突处理还需要更强的引导式流程。

## 迭代要求

| 领域 | 迭代要求 | 验收标准 |
|---|---|---|
| 启动与工作区 | 打开应用后必须能立即看到当前工作区、默认模型、网关状态和最近活动。 | 用户在 5 秒内能判断“我在哪、当前用什么、下一步做什么”。 |
| 对话流程 | 新建会话、继续会话、切换模型/提供商时，本地历史必须清晰保留并可追踪。 | 切换模型不会删除历史；界面明确说明保留范围和变化项。 |
| 提供商与模型 | 提供商创建、连接测试、模型创建、能力展示必须形成完整闭环。 | 无效 Base URL、无效 Key、不可用模型都要给出可操作的错误提示。 |
| 本地网关 | 网关启用、监听地址、端点列表、API Key 生成与撤销都要明确可见。 | 用户能确认网关是否启用、监听在哪、可调用哪些端点。 |
| 知识与数据 | 导入、解析、索引状态、错误回退和预留能力必须区分清楚。 | 文件状态、chunk 数、失败原因和重试动作都能在界面上找到。 |
| Tools / MCP / Agent | 注册、权限、审批、dry-run、预留执行边界必须严格分层。 | 任何危险执行都不能伪装成已完成；未授权能力必须保持不可执行状态。 |
| 日志与安全 | 请求日志、用量、错误诊断、审计、脱敏必须形成统一闭环。 | 每个错误状态至少提供“复制错误”和“打开日志”入口；密钥不得明文暴露。 |
| 导入导出与快照 | 配置导入、导出、备份、恢复、冲突处理必须可预测。 | 无效清单被拒绝；成功/失败结果清晰可见；默认会脱敏。 |

## 功能验收标准

- 用户能从首页一路走完“选模型 -> 发消息 -> 看结果 -> 查日志 -> 改配置”。
- 本地历史、配置快照、日志和诊断信息都能在界面上找到明确入口。
- 所有预留能力都必须显式标记为预留或计划中，不能冒充已实现。
- 任何失败状态都必须有下一步动作，而不是只有一条报错文本。
- 任何涉及密钥、审计、导出、危险操作的流程都必须有明确确认。
- 核心流程在功能上应保持稳定，不因切换模块而丢失上下文。

## 风险与非目标

- 本次不追求把所有高级能力一次性做完，先把主循环和边界做清楚。
- 不允许用“演示数据”掩盖真实流程缺失。
- 不把未完成的 Agent / 工作流能力包装成完整产品能力。
